`export async function evaluateAndEnqueueAlerts(admin): Promise<{ triggered: number; enqueued: number }>`:
1. Load active alerts joined to instrument display + account owner: `admin.from('price_alerts').select('id, account_id, instrument_id, direction, threshold, is_active, cooldown_minutes, last_triggered_at, instruments ( symbol, display_name, currency ), investment_accounts ( user_id )').eq('is_active', true)`. Map to the `AlertEvalRow` shape (05-03) — resolve `userId` from the joined account (the outbox recipient is a USER, resolved to chat_id at dispatch time — 05-RESEARCH-schema-outbox section 3). Keep the display fields alongside for message rendering.
2. If no active alerts → return `{ triggered: 0, enqueued: 0 }`.
3. Load `price_cache` for exactly those instrument_ids: `select instrument_id, price, fetch_error`; build a `Map<string, PriceSnapshot>` (`{ price, fetchError }`). Evaluation reads price_cache (NOT in-flight fetch results) so it also works when the dispatch route is hit standalone and matches exactly what the dashboard shows — one source of truth (05-RESEARCH-schema-outbox "Where").
4. `const triggered = evaluateAlerts(evalRows, priceMap, new Date());`
5. For EACH triggered alert, in order (enqueue-first — 05-RESEARCH-schema-outbox "enqueue-first rule"):
   a. Pre-render the message: `const text = buildPriceAlertMessage({ displaySymbol, direction, threshold, price, currency });` (payload.text fully rendered here keeps the dispatcher kind-agnostic).
   b. `await enqueueNotifications(admin, [{ userId, kind: 'price_alert', payload: { text, alertId, instrumentId, price, threshold }, dedupeKey: computeAlertDedupeKey(alertEvalRow, now) }]);` (ignoreDuplicates on dedupe_key is the crash-recovery backstop).
   c. THEN `await admin.from('price_alerts').update({ last_triggered_at: nowIso }).eq('id', alertId);` (plain update — the evaluator is the only writer of this column). Reversing this order would risk stamping the cooldown while losing the enqueue → swallowing the alert for a whole window; do NOT reverse it.
6. Return `{ triggered: triggered.length, enqueued: triggered.length }`.

Honest note to include as a comment: enqueue and the last_triggered_at update are two separate statements (supabase-js has no cross-call transaction — same boundary 04-04 faced); the dedupe_key unique index + enqueue-first ordering is the never-lost/never-duplicate backstop, not a true transaction.
