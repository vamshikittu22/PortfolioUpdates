  <files>src/lib/prices/get-portfolio-pnl.ts</files>
  <action>
This is the only place a Supabase client meets the pure P&L math тАФ it reads (never writes), so it can safely use the caller's cookie-bound server client (RLS-scoped reads are fine here; unlike 03-04's writes, no admin client needed).

```typescript
export interface PortfolioPnLResult {
  holdings: Array<Holding & HoldingPnL & { staleness: StalenessInfo; corporateActionFlag: boolean }>;
  portfolioTotal: PortfolioTotal;
  fxRate: number | null;
  fxFetchError: string | null;
}

export async function getPortfolioPnL(
  supabase: SupabaseClient,
  accountId: string,
  baseCurrency: 'INR' | 'USD'
): Promise<PortfolioPnLResult>
```

Steps:
1. `getHoldings(supabase, accountId)` from `@/lib/supabase/portfolio` (Phase 2, plan 02-04) тАФ reuse it, do not re-query transactions here.
2. `SELECT instrument_id, price, change_pct, updated_at, fetch_error, corporate_action_flag FROM price_cache WHERE instrument_id IN (...)` for the holdings' instrument ids.
3. `SELECT rate, updated_at, fetch_error FROM fx_cache WHERE pair = 'USD_INR'` (single row).
4. For each holding, call `calculateHoldingPnL` from `@/lib/prices/pnl-calculator` with `{quantity, avgCost, currency: holding.currency}` and the matched price_cache row's `price`/`change_pct` (or `null`/`null` if no row exists yet for that instrument тАФ the pending path).
5. Compute `staleness` per holding: derive a `StalenessInfo = { level: 'fresh' | 'stale' | 'very-stale' | 'error' | 'pending'; asOf: string | null }` тАФ `'error'` if that row's `fetch_error` is set, `'pending'` if no price_cache row/no price at all, else bucket `updated_at` age into `fresh` (&lt; 5 min... use a realistic bucket for a 3h schedule: &lt; 30 min), `stale` (&lt; 6h), `very-stale` (&gt;= 6h). Document the exact thresholds in a comment (tie them to the ~3h refresh interval from PRICE-02, not arbitrary numbers).
6. Call `calculatePortfolioTotals` with all holdings' PnL + `baseCurrency` + the fetched `fxRate` (or `1` if no FX row exists yet AND `baseCurrency` happens to match every holding's currency тАФ but if a non-base-currency holding exists and `fxRate` is null, this MUST surface as a visible "FX unavailable" state, not silently convert at `1`; document this edge case explicitly).
7. Return the combined result.
