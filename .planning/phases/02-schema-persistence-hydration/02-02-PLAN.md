<feature>
  <name>deriveHoldings — transaction ledger to holding aggregation</name>
  <files>src/lib/portfolio/derive-holdings.ts, scripts/derive-holdings-test.ts</files>
  <behavior>
Input: an array of transaction-shaped objects (see `Transaction` type below), each with `instrument_id`, `transaction_type` ('BUY'|'SELL'|'SPLIT'|'BONUS'), `quantity` (positive number), `price` (number or null), `transaction_date` (ISO date string).

Output: a `Map<string, { quantity: number; avgCost: number }>` (or equivalent array), keyed by `instrument_id`, computed by processing each instrument's transactions in chronological order (`transaction_date`, then `created_at` as tiebreaker if present) using a running weighted-average-cost method:

- Start `quantity = 0`, `costBasis = 0`.
- `BUY`: `costBasis += quantity_txn * price_txn`; `quantity += quantity_txn`.
- `SELL`: `avgCost = quantity > 0 ? costBasis / quantity : 0`; `costBasis -= quantity_txn * avgCost`; `quantity -= quantity_txn` (clamp at 0, never negative).
- `SPLIT` / `BONUS`: `quantity += quantity_txn`; `costBasis` UNCHANGED (no cash flow — this is what makes average cost dilute correctly instead of showing a false loss).
- Final `avgCost = quantity > 0 ? costBasis / quantity : 0`.
- Instruments whose derived quantity is 0 are either omitted from the result or included with `quantity: 0` — pick one and be consistent; document the choice in a comment (recommend: omit, since a zero-quantity row is not a "holding").

Cases (write these as the RED test cases in `scripts/derive-holdings-test.ts` BEFORE `derive-holdings.ts` has a real implementation — a stub that throws or returns `[]` unconditionally counts as RED):

1. Single BUY 10 @ ₹100 → `{ quantity: 10, avgCost: 100 }`.
2. BUY 10 @ ₹100, then SELL 4 @ ₹150 → `{ quantity: 6, avgCost: 100 }` (avgCost UNCHANGED by a partial sell — this is the PORT-04 correctness case).
3. BUY 10 @ ₹100, then SPLIT +10 (2-for-1) → `{ quantity: 20, avgCost: 50 }` (cost basis unchanged at 1000, avgCost halves — no false loss).
4. BUY 10 @ ₹100, then BONUS +5 → `{ quantity: 15, avgCost: 66.666... }` (cost basis unchanged at 1000).
5. BUY 10 @ ₹100, then SELL 10 @ ₹200 (full exit) → instrument omitted from result (or `quantity: 0` per the documented choice), no divide-by-zero throw.
6. Two different `instrument_id`s interleaved in one input array → each aggregates independently; verify neither leaks into the other's totals.
7. Out-of-order input array (SELL transaction object appears before its BUY in array order, but its `transaction_date` is later) → result must match the date-sorted order, not array order — proves the function sorts before reducing.

Use `node:assert/strict` for assertions (same dependency-free style as `scripts/rls-isolation-test.ts`), `console.log('PASS')` and `process.exit(0)` on success, throw / non-zero exit on any failed assertion. No new devDependency (no jest/vitest) — this matches the existing `test:rls` convention exactly.
  </behavior>
